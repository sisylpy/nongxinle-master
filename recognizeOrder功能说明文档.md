# recognizeOrder 图片识别下单功能详细说明文档

## 一、功能概述

`recognizeOrder` 是一个图片识别下单的核心功能接口，通过 OCR（光学字符识别）技术识别订单图片，然后使用 DeepSeek AI 进行智能解析和纠错，最终自动匹配商品并创建订单。

### 功能流程
```
图片上传 → OCR识别 → 文本预处理 → DeepSeek解析 → 商品匹配 → 订单创建
```

---

## 二、接口信息

### 2.1 接口定义
- **路径**: `/recognizeOrder`
- **方法**: `POST`
- **请求类型**: `application/json`
- **返回类型**: `R` (统一响应对象)

### 2.2 请求参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| ImageBase64 | String | 是 | 图片的Base64编码（支持带或不带data:image前缀） |
| depId | Integer | 否 | 部门ID（提供后自动查询商品并保存订单） |
| disId | Integer | 否 | 分销商ID（提供后自动查询商品并保存订单） |
| depFatherId | Integer | 否 | 部门父ID（不提供时自动从部门信息获取） |
| userId | Integer | 否 | 用户ID（默认-1） |

### 2.3 返回数据结构

#### 成功返回（提供了depId和disId）
```json
{
  "code": 0,
  "msg": "success",
  "data": [
    {
      "nxDepartmentOrdersId": 订单ID,
      "nxDoGoodsName": "商品名称",
      "nxDoQuantity": "数量",
      "nxDoStandard": "规格",
      "nxDoStatus": 订单状态,
      "nxDoDisGoodsId": 商品ID,
      "nxDistributerGoodsEntityList": [推荐商品列表],
      ...
    }
  ]
}
```

#### 成功返回（未提供depId和disId）
```json
{
  "code": 0,
  "msg": "success",
  "ocrText": "OCR识别的文本内容",
  "items": [
    {
      "name": "商品名称（纠错后）",
      "rawName": "原始商品名称（OCR识别）",
      "quantity": "数量",
      "spec": "规格",
      ...
    }
  ],
  "parsedResult": "DeepSeek原始解析结果"
}
```

## 三、核心功能流程详解

### 3.1 图片预处理阶段

#### 3.1.1 Base64处理
- 检查图片数据是否为空
- 处理Base64前缀（自动去除`data:image`前缀）
- 验证图片大小（限制10MB，Base64编码后）

#### 3.1.2 OCR识别
- 使用腾讯云OCR服务进行文字识别
- **重试机制**: 最多重试3次，针对网络错误自动重试
- **重试条件**: IOException、unexpected end of stream、timeout、Connection错误
- **重试间隔**: 递增等待时间（1秒、2秒、3秒）

### 3.2 OCR结果预处理阶段

#### 3.2.1 文本过滤
过滤以下类型的文本：
1. **噪声文本** (`isNoise`方法)
   - 空字符串
   - 纯横线（如`----`）
   - 单个字母
   - 纯符号（不包含中文、字母、数字）

2. **无关文本** (`isIrrelevantText`方法)
   - 页码、页眉页脚
   - 日期时间格式
   - 其他非订单相关的文本

3. **低置信度文本**
   - 置信度低于50%的文本（商品名即使置信度较低也会保留）

#### 3.2.2 坐标聚类
- **行聚类** (`groupByRows`方法)
  - 按Y坐标聚类，阈值：25像素
  - 同一行的文本按X坐标排序
  - 构建按行组织的OCR文本

#### 3.2.3 文本清洗
- 使用`normalizeOcrLine`方法清洗每行文本
- 去除多余空格、特殊字符等

### 3.3 DeepSeek AI解析阶段

#### 3.3.1 API调用
- **模型**: `deepseek-chat`
- **超时设置**: 
  - 连接超时：30秒
  - Socket超时：120秒
  - 连接池超时：30秒

#### 3.3.2 解析规则
DeepSeek按照以下规则解析订单：

**P0级规则（最高优先级）**:
- 绝对不允许吞掉任何一行订单条目
- 数量/单位必须严格保留
- 商品名称必须纠错（如"气椒"→"青椒"）
- 同一商品多组数量必须相加（如"10斤+10斤"→"20斤"）

**P1级规则**:
- 仅对商品名称进行纠错
- 数量和规格必须严格按原始文本解析
- 纠错优先按字形相似度（而非读音）

#### 3.3.3 返回结果解析
支持两种格式：
1. **JSON数组格式**: 直接是商品数组
2. **JSON对象格式**: 包含`orderItems`或`items`字段

每个商品包含字段：
- `name`: 纠错后的商品名称
- `rawName`: OCR原始识别的商品名称
- `quantity`: 数量
- `spec`: 规格
- `standardWeight`: 标准重量
- `note`: 备注
- `isNotice`: 提示信息

#### 3.3.4 本地后处理
1. **手写体识别**: 规格"h"转换为"斤"
2. **数量拆分**: 如果规格为空且数量是3位数，拆分第一位作为数量，后两位作为规格"个"
3. **字段映射**: 
   - `quantity` → `qty`
   - `spec` → `unit`
   - `note` → `remark`

### 3.4 商品匹配与订单创建阶段

**前提条件**: 必须提供`depId`和`disId`参数

#### 3.4.1 订单实体初始化
为每个解析出的商品创建基础订单实体：
- 部门ID、分销商ID、用户ID
- 商品名称（使用`name`，纠错后的名称）
- 数量、规格、备注
- 原始商品名称（使用`rawName`，用于训练数据）

#### 3.4.2 强查询匹配（使用rawName）

**查询顺序**（任一匹配成功即停止）：

1. **1级优先匹配：部门商品历史记录**
   - 查询条件：部门ID + rawName + 规格
   - 匹配要求：完全匹配（结果数量=1）
   - 订单状态：
     - 如果数量或规格已改变：状态=-2（待修正）
     - 否则：状态=0（已完成）

2. **2级优先匹配：商品库名称查询**
   - 查询条件：分销商ID + rawName + 规格
   - 匹配要求：完全匹配（结果数量=1）
   - 订单状态：同上

3. **3级优先匹配：商品库别名查询**
   - 查询条件：分销商ID + name（纠错后的名称）+ 规格
   - 匹配要求：完全匹配（结果数量=1）
   - 订单状态：同上

4. **4级优先匹配：训练数据查询**
   - 查询条件：部门ID + rawName + 有商品ID
   - 匹配要求：找到训练数据且有关联商品ID
   - 特殊处理：优先使用训练数据的`final_goods_name`（如果存在）
   - 订单状态：同上

#### 3.4.3 弱查询匹配（使用name）

**触发条件**: 强查询全部失败后

**流程**:
1. **创建训练数据**
   - `original_goods_name` = rawName
   - `deepseek_recommended_name` = name
   - 保存训练数据，等待人工标注

2. **弱查询匹配**（任一匹配成功即停止）:

   a. **1级弱查询：部门商品历史记录**
      - 查询条件：部门ID + name + 规格
      - 订单状态：统一为-2（待修正）
      - 操作：保存订单、关联训练数据、添加推荐商品

   b. **2级弱查询：商品库名称查询**
      - 查询条件：分销商ID + name + 规格
      - 订单状态：统一为-2（待修正）
      - 操作：保存订单、关联训练数据、添加推荐商品

   c. **3级弱查询：训练数据查询**
      - 查询条件：部门ID + name + 有商品ID
      - 订单状态：统一为-2（待修正）
      - 操作：保存订单、关联训练数据、添加推荐商品

3. **模糊搜索**（弱查询全部失败后）
   - 调用`searchAndSaveOrdersFromOcr`方法
   - 进行模糊商品搜索
   - 订单状态：-2（待修正）
   - 关联训练数据ID

#### 3.4.4 订单状态说明

| 状态值 | 含义 | 使用场景 |
|--------|------|----------|
| 0 | 已完成 | 强查询匹配成功，且数量/规格未改变 |
| -2 | 待修正 | 强查询匹配成功但数量/规格已改变，或弱查询匹配成功，或模糊搜索 |

#### 3.4.5 推荐商品机制

**触发条件**: 
- 弱查询匹配成功
- 模糊搜索找到商品

**功能**: 
- 调用`addCommentsGoodsForOrder`方法
- 为订单添加相似商品推荐列表
- 不改变订单状态

## 四、辅助方法说明

### 4.1 订单状态设置方法

**方法名**: `setOrderStatusByChangeFlag`

**功能**: 根据数量和规格是否改变设置订单状态

**参数**:
- `orderBasic`: 订单实体
- `specIsChange`: 规格是否改变
- `quantityIsChange`: 数量是否改变

**逻辑**:
- 如果数量或规格已改变 → 状态=-2（待修正）
- 否则 → 状态=0（已完成）

### 4.2 强查询订单保存方法

**方法名**: `saveStrongQueryOrderAndConvert`

**功能**: 保存强查询匹配的订单并转换为响应DTO

**参数**:
- `orderBasic`: 订单实体
- `distributerGoodsEntity`: 分销商商品实体
- `note`: 备注
- `originalIndex`: 原始索引（保持顺序）
- `responseMap`: 响应Map

**操作**:
1. 保存订单（`saveOrderWithGoods`）
2. 转换为响应DTO（`convertOrderToResponseDTO`）
3. 存入responseMap

### 4.3 弱查询订单保存方法

**方法名**: `saveWeakQueryOrderWithTrainingData`

**功能**: 保存弱查询匹配的订单（关联训练数据、添加推荐商品）并转换为响应DTO

**参数**:
- `orderBasic`: 订单实体
- `distributerGoodsEntity`: 分销商商品实体
- `trainingData`: 训练数据实体
- `note`: 备注
- `originalIndex`: 原始索引（保持顺序）
- `responseMap`: 响应Map

**操作**:
1. 设置订单状态为-2
2. 保存订单（`saveOrderWithGoods`）
3. 关联训练数据ID（`setNxDoTrainingDataId`）
4. 更新订单（`update`）
5. 添加推荐商品（`addCommentsGoodsForOrder`）
6. 转换为响应DTO（`convertOrderToResponseDTO`）
7. 存入responseMap

### 4.4 其他辅助方法

- `createOrQueryTrainingData`: 创建或查询训练数据
- `convertOrderToResponseDTO`: 订单实体转响应DTO
- `groupByRows`: OCR框按Y坐标聚类成行
- `normalizeOcrLine`: 清洗OCR文本行
- `isNoise`: 判断是否为噪声文本
- `isIrrelevantText`: 判断是否为无关文本
- `isValidParseResult`: 验证解析结果有效性

## 五、错误处理

### 5.1 OCR识别错误
- **图片为空**: 返回"图片数据不能为空"
- **图片过大**: 返回"图片大小超过限制，最大支持 10MB（Base64 编码后）"
- **未识别到文本**: 返回"未识别到任何文本"
- **资源包耗尽**: 返回友好的提示信息
- **认证失败**: 返回"认证失败，请检查 SecretId 和 SecretKey 配置是否正确"
- **网络错误**: 自动重试最多3次

### 5.2 DeepSeek解析错误
- **API调用失败**: 返回"订单解析失败：DeepSeek API 调用失败"
- **解析结果无效**: 返回"订单解析失败：DeepSeek 解析结果无效"
- **解析结果为空**: 返回"订单解析失败：DeepSeek 解析结果无效"

### 5.3 订单处理错误
- **商品不存在**: 抛出RuntimeException（训练数据关联的商品ID不存在）
- **处理订单失败**: 即使失败也返回OCR识别结果和解析结果，便于调试

## 六、关键设计要点

### 6.1 强查询 vs 弱查询
- **强查询**: 使用OCR原始识别的`rawName`，匹配度高，订单状态可能是0或-2
- **弱查询**: 使用DeepSeek纠错后的`name`，匹配度较低，订单状态统一为-2

### 6.2 训练数据机制
- **创建时机**: 强查询全部失败后
- **用途**: 记录OCR原始识别结果和AI推荐结果，等待人工标注
- **关联**: 弱查询和模糊搜索的订单都会关联训练数据ID

### 6.3 订单状态设计
- **0（已完成）**: 强查询匹配成功，且数量/规格未改变，可直接使用
- **-2（待修正）**: 需要人工确认，包括：
  - 强查询匹配但数量/规格已改变
  - 弱查询匹配成功
  - 模糊搜索找到商品

### 6.4 推荐商品机制
- **目的**: 为不确定的订单提供相似商品选择
- **触发**: 弱查询和模糊搜索找到商品后
- **不影响**: 不改变订单状态和已匹配的商品

### 6.5 代码优化
- **提取重复代码**: 将订单状态设置、订单保存等重复逻辑提取为私有方法
- **保持功能不变**: 所有优化都保持原有业务逻辑不变

## 八、注意事项

1. **图片格式**: 支持常见图片格式（jpg、png等），Base64编码
2. **图片大小**: 限制10MB（Base64编码后）
3. **OCR服务**: 依赖腾讯云OCR服务，需要配置SecretId和SecretKey
4. **DeepSeek服务**: 依赖DeepSeek API，需要配置API URL和密钥
5. **订单创建**: 必须提供depId和disId才会创建订单
6. **训练数据**: 强查询失败时会自动创建训练数据，等待人工标注
7. **推荐商品**: 弱查询和模糊搜索的订单会自动添加推荐商品列表

**核心优势**:
1. 多级匹配机制，提高匹配准确率
